    Errors, Exceptions, & Warnings

    Errors

  • Errors: something goes wrong in a program
    1. System error: there’s a problem with the system or external devices with which the program is interacting.
    2. Programmer error: the program contains incorrect syntax or faulty logic; it could even be as simple as a typo
    3. User error: the user has entered data incorrectly, which the program is unable to handle
  • try to limit user errors by predicting any possible interactions that may throw an error, & ensure they are dealt with in a way that doesn’t negatively affect the user experience

    • Exceptions

  • Exceptions: an error that produces a return value that can then be used by the program to deal with the error
    1. Stack Trace: produced by an exception; a sequence of functions or method calls that lead to the point where the error occurred

    Warnings

  • Warnings: can occur if there’s an error in the code that isn't enough to cause the program to crash
  • Strict mode: produces more exceptions and warnings and prohibits the use of some deprecated features; considers coding practices that were previously accepted as just being 'poor style' as actual errors
    1. Strict mode simply requires the following string to be added to the first line of a JavaScript file:
    2. 'use strict';
  • Linting Tools: can be used to test the quality of JavaScript code, beyond simply using strict mode; designed to highlight any sloppy programming practices or syntax errors; will complain if certain style conventions are not followed
  • Feature Detection: recommended way to determine browser support for a feature; done using an if statement to check whether an object or method exists before trying to actually call the method

    • Debugging in the Browser

  • Debugging: the process of finding out where bugs occur in the code and then dealing with them
  • Trusty Alert: most basic form of debugging is to use the alert() method to show a dialog at certain points in the code; stops a program from running until OK is clicked
  • Using the console: can be used to log the value of variables at different stages of the program; will not stop the program
    1. console.log() -> log the position in the function as well as the value of the variable
    2. console.trace() -> log an interactive stack trace in the console
    3. SitePoint discusses other less well known console methods
  • Breakpoint: a debugging tool which pauses the code at certain points
    1. "debugger" keyword: will create a breakpoint in your code that will pause the execution of the code and allow you to see where the program is currently up to
  • Error Objects: can be created by the host environment when an exception occurs; can be created in the code using a constructor function =>
    const error = new Error();
    const error = new Error('Oops, something went wrong');
    1. 7 More Error Objects:
    2. EvalError: not used in the current ECMAScript specification and only retained for backwards compatibility. It was used to identify errors when using the global eval() function.
    3. RangeError: thrown when a number is outside an allowable range of values
    4. ReferenceError: thrown when a reference is made to an item that doesn’t exist
    5. SyntaxError: thrown when there’s an error in the code’s syntax
    6. TypeError: thrown when there’s an error in the type of value used
    7. URIError: thrown when there’s a problem encoding or decoding the URI
    8. InternalError: a non-standard error that is thrown when an error occurs in the JavaScript engine. A common cause of this too much recursion.
  • can also be used as constructors to create custom error objects
    const error = new TypeError('You need to use numbers in this function');
  • have a number of properties; only properties that are generally safe to use are:
    1. name: returns the name of the error constructor function used as a string
    2. message: returns a description of the error and should be provided as an argument to the Error constructor function
    3. stack: will return a stack trace for that error

    Throwing Exceptions

  • Throw statement: can be applied to any JavaScript expression, causing the execution of the program to stop
  • best practice, however, to throw an error object:
    throw new Error('Something has gone badly wrong!');

    • Exception Handling

  • Any errors can be hidden from users, but still identified. We can then deal with the error appropriately ― perhaps even ignore it ― and keep the program running.
  • try block: will run the code inside the block as normal, but if an exception occurs it will pass the error object that is thrown onto a catch block
  • catch block: code inside will only run if an exception is thrown inside the try block; the error object is automatically passed as a parameter to the catch block. This allows us to query the error name, message and stack properties, and deal with it appropriately
  • finally block: can be added after a catch block. This will always be executed after the try or catch block, regardless of whether an exception occurred or not

    • Tests

  • Writing good tests means your code will be less brittle as it develops, and any errors will be identified early on.
    1. can simply be a function that tests a piece of code runs as it should
    2. This is in no way a thorough test of the function – it might just be a fluke.

    Test-Driven Development (TDD)

  • TDD: the process of writing tests before any actual code
  • The next step is to write some code to make the tests pass.
  • Next, the code is refactored to make it faster, more readable, and remove any repetition.
  • The code is continually tested at each stage to make sure it continues to work
    1. Write tests (that initially fail)
    2. Write code to pass the tests
    3. Refactor the code
    4. Test refactored code
    5. Write more tests for new features
  • Testing Frameworks: provide a structure to write meaningful tests and then run them

    • Chapter Summary

  • Bugs are unavoidable in code, and it’s best to find them early rather than later.
  • JavaScript can be put into strict mode using the string "use strict" . This can be used in a whole file or just a single function.
  • Linting tools can be used to ensure your code follows good practice and conventions.
  • Feature detection can check whether a method is supported before calling it, helping to avoid an exception being thrown.
  • The console and browser’s built-in debugging tool can be used to interactively find and fix bugs in code.
  • Exceptions can be thrown using the throw statement.
  • An error object is created when an exception occurs.
  • Any code placed inside a try block will pass any error objects to a catch block when an exception occurs. Any code inside a finally block will run if an exception does or does not occur.
  • Test-driven development is the practice of writing tests that fail, then writing the code that passes the test, then refactoring the code every time a new feature is implemented.
  • The Jest framework can be used to test your code.
